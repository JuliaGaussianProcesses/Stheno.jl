154 """
155 Thunk(()->v)
156 A thunk is a deferred computation.
157 It wraps a zero argument closure that when invoked returns a tangent.
158 `@thunk(v)` is a macro that expands into `Thunk(()->v)`.
159
160 To evaluate the wrapped closure, call [`unthunk`](@ref) which is a no-op when the
161 argument is not a `Thunk`.
162
163 ```jldoctest
164 julia> t = @thunk(3)
165 Thunk(var"#4#5"())
166
167 julia> unthunk(t)
168 3
169 ```
170
171 ### When to `@thunk`?
172 When writing `rrule`s (and to a lesser exent `frule`s), it is important to `@thunk`
173 appropriately.
174 Propagation rules that return multiple derivatives may not have all deriviatives used.
175 By `@thunk`ing the work required for each derivative, they then compute only what is needed.
176
177 #### How do thunks prevent work?
178 If we have `res = pullback(...) = @thunk(f(x)), @thunk(g(x))`
179 then if we did `dx + res[1]` then only `f(x)` would be evaluated, not `g(x)`.
180 Also if we did `ZeroTangent() * res[1]` then the result would be `ZeroTangent()` and `f(x)` would not be evaluated.
181
182 #### So why not thunk everything?
183 `@thunk` creates a closure over the expression, which (effectively) creates a `struct`
184 with a field for each variable used in the expression, and call overloaded.
185
186 Do not use `@thunk` if this would be equal or more work than actually evaluating the expression itself.
187 This is commonly the case for scalar operators.
188
189 For more details see the manual section [on using thunks effectively](http://www.juliadiff.org/ChainRulesCore.jl/dev/writing_good_rules.html#Use-Thunks-appropriately-1)
190 """

192 f::F
193 end
