10 has_chain_rrule(T)
11
12 For a type-tuple `T` e.g. `Tuple{typeof(f), Int, Float64}`, checks if there is a `rrule` defined for it.
13 Excluding the generic fallback.
14 The first return value is `true` if the `rrule` exists, `false` otherwise.
15 If it does not, then the second argument is a list of edges to attach to the CodeInfo for a generated function,
16 such that if a suitable rule is defined later, the generated function will recompile.
17 """
18 function has_chain_rrule(T)
19 config_T, arg_Ts = Iterators.peel(T.parameters)
20 configured_rrule_m = meta(Tuple{typeof(rrule), config_T, arg_Ts...})
21 if _is_rrule_redispatcher(configured_rrule_m.method)
22 # The config is not being used:
23 # it is being redispatched without config, so we need the method it redispatches to
24 rrule_m = meta(Tuple{typeof(rrule), arg_Ts...})
25 # Thus any no_rrule that might apply must also not have a config because if there was a
26 # no_rrule with a config that applied then there would also be a rrule with config that applied
27 no_rrule_m = meta(Tuple{typeof(ChainRulesCore.no_rrule), arg_Ts...})
28 else
29 # Not being redispatched: it does have a config
30 rrule_m = configured_rrule_m
31 # Thus any no_rrule that might apply must also have a config because if it applied
32 # it will be identical, and if it doesn't we don't care what it is.
33 no_rrule_m = meta(Tuple{typeof(ChainRulesCore.no_rrule), config_T, arg_Ts...})
34 end

36 # To understand why we only need to check if the sigs match between no_rrule_m and rrule_m
37 # in order to decide if to use, one must consider the following facts:
38 # - for every method in `no_rrule` there is a identical one in `rrule` that returns nothing
39 # - this includes the general fallback `rrule(::Any...)=nothing`.
40 # - a configured rrule/no_rrule is always more specific than a otherwise equivalent unconfigured rrule/no_rrule
41 #
42 # Consider the following truth table, for what can occur:
43 # rrule: fallback, no_rrule: fallback => matches => do not use rrule.
44 # rrule: specific, no_rrule: fallback => !matches => do use rrule, as haven't opted out.
45 # rrule: fallback, no_rrule: specific => IMPOSSIBLE, every no_rule is identical to some rrule
46 # rrule: specific, no_rrule: specific => matches => do not use rrule as opted out
47 # rrule: specific, no_rrule: general => !matches => do use rrule as a more specific rrule takes preciedent over more general opted out
48 # rrule: general , no_rrule: specific => IMPOSSIBLE, every no_rule us identical to some rrule so can't have a more general rrule being hit, as the specific one would hit first
49 #
50 # Note that the fallback cases are the same outcome as the general cases as fallback is just most general.
51 # It can be seen that checking if it matches is the correct way to decide if we should use the rrule or not.
52
53
54 do_not_use_rrule = matching_cr_sig(no_rrule_m, rrule_m)
55 if do_not_use_rrule
56 # Return instance for configured_rrule_m as that will be invalidated
57 # directly if configured rule added, or indirectly if unconfigured rule added
58 # Do not need an edge for `no_rrule` as no addition of methods to that can cause this
59 # decision to need to be revisited (only changes to `rrule`), since we are already not
60 # using the rrule, so not using more rules wouldn't change anything.
61 return false, configured_rrule_m.instance
62 else
63 # Otherwise found a rrule, no need to add any edges for `rrule`, as it will generate
64 # code with natural edges if a new method is defined there.
65 # We also do not need an edge to `no_rrule`, as any time a method is added to `no_rrule`
66 # a corresponding method is added to `rrule` (to return `nothing`), thus we will already
67 # be revisiting this decision when a new opt-out is added.
68 return true, nothing
69 end
70 end
